# 重构项目协作规则

## 背景

当前仓库 `web-ele` 是一个重构项目，不是从零开始的新项目。

涉及三个明确的参照源：

1. 当前正在运行的业务项目代码：
   `/Users/liujun/workspace/shop/shop/frontend/admin`
2. 当前重构时参考的 Vben 项目：
   `/Users/liujun/workspace/github/vue-vben-admin/apps/web-ele`
3. 官方组件文档：
   `https://doc.vben.pro/components/introduction.html`

本规则用于约束后续所有重构、迁移、页面补齐、交互调整和提交行为，避免继续出现“按模板重写”或“脱离现有业务结构”的问题。

## 目标

本项目的目标不是复刻官方 demo，也不是保留旧项目的所有实现细节，而是：

1. 以 Vben 的布局能力、组件能力、路由能力承载现有业务系统。
2. 以旧业务项目为真实业务来源，保留其接口、菜单、页面职责和核心交互。
3. 在不破坏当前可运行状态的前提下，逐步完成页面、接口、权限和样式的重构迁移。

## 核心判断原则

后续开发一律按下面的优先级做判断：

1. 业务真实需求优先于模板默认实现。
2. 当前运行项目的业务结构优先于参考项目的演示结构。
3. 参考项目只提供框架组织方式、布局方式和基础交互参考，不直接当业务真相。
4. 官方组件文档只说明“怎么用框架组件”，不代表“必须使用框架组件”。

## 三个来源的职责边界

### 1. 运行项目 `/Users/liujun/workspace/shop/shop/frontend/admin`

这个项目是业务真相来源，后续以下内容必须以它为准：

1. 页面目录与业务模块划分
2. 菜单语义与页面职责
3. API 路径、参数、返回结构
4. 权限点、按钮权限、状态值、字典值
5. 页面上的真实表单项、表格列、详情结构
6. 登录、验证码、用户信息、头像、动态菜单等真实链路

结论：
如果运行项目和参考项目不一致，优先服从运行项目。

### 2. 参考项目 `/Users/liujun/workspace/github/vue-vben-admin/apps/web-ele`

这个项目是框架实现参考，不是业务来源。后续以下内容可以参考它：

1. 目录组织方式
2. 布局接入方式
3. 路由守卫组织方式
4. 登录页、错误页、锁屏、全局 loading 等框架交互
5. Vben 组件接入方式
6. 组件适配层、表单适配层、基础偏好设置的写法

结论：
参考项目只能借鉴“框架接法”和“通用交互模式”，不能覆盖业务结构。

### 3. 官方文档 `https://doc.vben.pro/components/introduction.html`

官方文档中的关键原则是：

- 文档介绍的是框架组件的使用方法。
- 如果现有封装不合适，可以直接使用原生组件，或者自行封装适合的组件。
- 框架组件不是束缚，是否使用取决于实际需求。

基于该原则，后续统一要求：

1. 不强制把旧项目所有组件都改成 Vben 组件。
2. 能复用现有业务组件时，优先复用。
3. Vben 组件只在确实能降低维护成本、提升一致性时使用。
4. 当 Vben 组件不适配业务场景时，允许直接使用 Element Plus、原生组件或项目内封装组件。

## 重构硬规则

### 目录与模块

1. 页面目录命名优先对齐当前运行项目。
2. 业务页面不要再按 `_core`、`demos`、示例目录扩散。
3. 非业务演示页面一律清理，不重新引入。
4. 新增页面时，优先补业务壳页面，不先做演示页。

### 路由与菜单

1. 动态菜单以服务端返回为准。
2. 菜单组装时，只做最小必要转换：
   - `Layout` 转布局组件
   - 框架保留字段做必要归一化
3. 不擅自把后端菜单重新映射成另一套路由语义。
4. 页面不存在时，优先落到统一兜底页，不伪造业务实现。
5. 路由名必须稳定唯一，不能因空 `name` 造成冲突。

### 接口与数据

1. 一律以真实接口为准，不再引入 mock 兜底链路。
2. 接口适配优先在 `api` 层解决，不把后端结构差异扩散到页面层。
3. 认证失败、刷新 token、用户信息、菜单获取必须走真实接口。
4. 业务码和 HTTP 状态码要分开处理，不能只假设后端返回 HTTP 401。

### 组件使用

1. 先判断业务是否已有稳定组件。
2. 业务已有稳定组件时，优先复用，不强行替换。
3. 表单、抽屉、弹窗、表格等基础设施，可优先借助 Vben 能力。
4. 当业务交互复杂且 Vben 封装不贴合时，允许直接使用 Element Plus 或项目组件。

### 样式与视觉

1. 优先保持当前项目已确定的视觉方向，不回退到模板默认示例态。
2. 登录、错误页、锁屏、首屏 loading 可参考参考项目，但不能影响业务链路。
3. 能用现有布局能力实现的，不重复造一套全局样式体系。

### 文案与国际化

1. 当前项目已明确移除国际化运行时。
2. 后续新增页面直接写中文文案，不再引入 `$t()`、`@vben/locales`、`#/locales`。
3. 若未来要恢复国际化，必须单独立项，不在当前重构中顺手恢复。

### 删除与保留

1. 删除页面前，先判断它是否属于错误演示页、无用模板页或已废弃业务页。
2. 用户明确说明“删除是正确的”后，按删除状态继续开发，不擅自恢复。
3. 对于不确定是否该删的页面，先确认再动。

## 开发执行规则

### 改动前

1. 先判断需求来自业务项目、参考项目，还是官方文档。
2. 明确“谁是事实来源”后再开始改。
3. 如果三个来源冲突，必须在改动前说明采用哪一个。

### 改动时

1. 优先做最小闭环改动。
2. 优先保证可运行、可登录、可进入首页、可获取菜单。
3. 批量迁移时，先打通链路，再逐页补细节。
4. 不为了追求“更像官方模板”而牺牲现有业务结构。

### 改动后

1. 至少验证构建或启动链路。
2. 涉及登录、菜单、权限、路由时，要验证核心跳转链路。
3. 如果只是参考项目中的交互优化，必须确认没有破坏业务接口和菜单逻辑。

## 沟通规则

后续讨论和执行时，默认使用下面的话术和判断方式：

1. 说“当前项目”时，指 `web-ele` 重构仓库。
2. 说“运行项目”时，指 `/Users/liujun/workspace/shop/shop/frontend/admin`。
3. 说“参考项目”时，指 `/Users/liujun/workspace/github/vue-vben-admin/apps/web-ele`。
4. 说“官方文档”时，指 `https://doc.vben.pro/components/introduction.html`。

当用户要求“参照项目处理”时，默认含义是：

1. 参照参考项目的框架实现方式；
2. 同时保持运行项目的业务结构不变；
3. 不直接照搬 demo 页面、目录或假数据。

## 结论

后续所有开发统一按一句话执行：

以运行项目为业务真相，以参考项目为框架实现参考，以官方文档为组件使用边界说明，做最小必要重构，不做脱离业务的模板化改写。
