# Codex 规则

## 文档优先级
- 开始任何分析、设计、修改前，优先查看 `docs/` 目录下的规则文档。
- 如果 `docs/` 下存在与当前任务相关的文档，默认将其视为当前仓库的一线执行规则，优先级高于通用习惯。
- 当 `AGENTS.md` 与 `docs/` 中的项目规则不冲突时，同时遵守；如果出现任务级别的具体约束，以 `docs/` 中对应文档为先。
- 当前重构项目默认优先参考：
  - [docs/01-重构项目协作规则.md](/Users/liujun/workspace/github/web-ele/docs/01-重构项目协作规则.md)

## 提交与文档约定
- 提交流程固定为以下顺序：
  1再检查并更新 `README.md`，确保文档与代码行为一致。
  2最后执行提交与推送，并将 `README.md` 改动与本次代码改动一起提交。
- 用户要求“提交”时，默认执行完整发布动作：`git commit` + `git push`。
- 用户要求“提交”时，默认先执行 `git add -A`，将未暂存与未跟踪文件一并加入本次提交（遵循 `.gitignore` 与用户明确排除的文件）。
- 未明确指定分支时，推送当前分支到同名远程分支。
- `git commit -m` 信息默认使用中文，简洁描述本次变更。
- 若用户未指定提交信息，按变更内容自动生成中文提交信息。

## 思考过程与记录
- 设计、边界和交互取舍优先记录到 `docs/`，不要只留在提交信息里。
- 新增设计文档建议使用“版本号前缀 + 中文主题”的命名方式，并在文档开头写清背景、目标、约束、结论。
- 如果某次修改明显改变交互或数据结构，应补一份简短记录，说明“为什么改、改了什么、兼容性影响是什么”。
- 当用户提出“迭代需求”时，先根据需求内容在 `docs/` 新建新的版本迭代设计文档并给出确认请求，用户确认后再进入开发与改动。
